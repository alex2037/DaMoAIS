# Модуль 4. Инфраструктура данных и RAG-контуры

## Место в дисциплине и связь со сквозным проектом

Модуль соответствует разделу 4 РПД и лекциям 5–6. К этому моменту сквозной проект уже имеет работающий inference-сервис в Kubernetes (итог ЛР2–ЛР3). Модуль добавляет к сервису слой данных: объектное хранилище моделей и артефактов, векторную базу для семантического поиска, кэш горячих ответов и реляционную БД метаданных с резервированием. Результат ЛР4 — собранный RAG-контур (MinIO + Qdrant + Redis) с разграничением доступа read-only/read-write — становится этапом 3 сквозного проекта («Данные/RAG и поставка») и в дальнейшем подключается к CI/CD-поставке (ЛР5) и мониторингу (ЛР6).

Проверяемые индикаторы КРМ (версия 3.0): **LC-5.2** (управление данными с нужным уровнем доступа, контроля качества, резервирования и скорости запросов) и **BD-5.1** (выбор вспомогательных технологических решений для единого стека данных). Целевой уровень модуля — **средний (С)**.

## Содержание лекции

- **Объектное хранилище MinIO** — S3-совместимый API; buckets, версионирование объектов, lifecycle-политики; клиент `mc`; хранение датасетов, моделей и артефактов инференса.
- **Версионирование данных** — DVC (`.dvc`-файлы, remote в MinIO/S3, связка с git-историей) и LakeFS (branch/commit/merge поверх объектного хранилища); воспроизводимость датасетов и моделей.
- **Feature store Feast** — offline/online store, materialization, единый источник фичей для обучения и инференса, борьба с training/serving skew.
- **Кэширование** — Redis и Memcached; кэш эмбеддингов и результатов retrieval; TTL и eviction-политики (LRU/LFU); влияние на p95-latency.
- **Реляционный слой и резервирование** — PostgreSQL, потоковая репликация (streaming replication, `wal_level=replica`, primary/hot standby); бэкапы (`pg_dump`, `pg_basebackup`), snapshot'ы, PITR по WAL.
- **Разграничение доступа** — RBAC/IAM/ACL; роли и GRANT в PostgreSQL, IAM-политики MinIO, API-ключи Qdrant; контуры read-only для инференса и read-write для ingestion/обучения; multi-tenant доступ и квоты.
- **Векторные БД и индексы** — Milvus/Qdrant/Pinecone; сравнение по критериям; типы индексов (HNSW, IVF), метрики расстояния (cosine, dot, L2), фильтрация по метаданным; trade-off latency/качество (полнота retrieval).
- **Эмбеддинги** — модели эмбеддингов, размерность вектора, нормализация; связь качества retrieval с выбором модели.
- **RAG-контур** — этап ingestion (chunking → embedding → upsert) и этап retrieval (query → embedding → top-k → сборка контекста → генерация); read-only контур инференса против read-write контура наполнения.

## Экспресс-тест текущего контроля

**Вопрос 1.** Какой протокол/API реализует MinIO, что позволяет использовать его как замену облачному хранилищу без изменения кода приложения?
- A) NFS
- B) S3-совместимый API
- C) POSIX filesystem
- D) JDBC

*Верный ответ: B.*

**Вопрос 2.** В RAG-контуре ingestion-конвейер выполняет операции в правильном порядке:
- A) retrieval → генерация → chunking
- B) embedding → поиск top-k → ответ
- C) chunking → embedding → upsert в векторную БД
- D) upsert → chunking → нормализация

*Верный ответ: C.*

**Вопрос 3.** Для доступа inference-сервиса к векторной БД и метаданным в production-контуре следует выдать:
- A) права read-write, чтобы сервис мог дообучаться на лету
- B) права суперпользователя для скорости
- C) права read-only, а запись оставить только конвейеру ingestion
- D) анонимный доступ без аутентификации

*Верный ответ: C.*

**Вопрос 4.** Что обеспечивает потоковая репликация PostgreSQL (primary → standby)?
- A) шардирование данных по нескольким узлам
- B) горячий резерв и отказоустойчивость чтения за счёт передачи WAL на реплику
- C) автоматическое версионирование датасетов
- D) кэширование эмбеддингов в памяти

*Верный ответ: B.*

**Вопрос 5.** Какой индекс векторной БД (Qdrant/Milvus) обеспечивает приближённый поиск ближайших соседей с хорошим балансом скорости и полноты?
- A) B-tree
- B) HNSW
- C) hash-индекс
- D) bitmap-индекс

*Верный ответ: B.*
