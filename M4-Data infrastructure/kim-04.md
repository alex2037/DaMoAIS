# КИМ модуля 4. Инфраструктура данных и RAG-контуры — ЛР4: Сборка data/RAG-контура с разграничением доступа

## Назначение

КИМ проверяет способность обучающегося построить слой данных для промышленного ИИ-сервиса: объектное хранилище артефактов (MinIO), векторную базу для семантического поиска (Qdrant), кэш горячих ответов (Redis) и реляционную БД метаданных с резервированием (PostgreSQL), связав их в работающий RAG-контур с разграничением доступа read-only/read-write. Результат становится этапом 3 сквозного проекта («Данные/RAG и поставка») и в дальнейшем поставляется через CI/CD (ЛР5) и наблюдается через мониторинг (ЛР6).

## Условия выполнения

- **Среда.** Linux/macOS/WSL2; Docker и Docker Compose (либо кластер Kubernetes из ЛР3 для продвинутого уровня). Python 3.11+.
- **Инструменты (реальные, без выдуманных версий).** MinIO + клиент `mc`; Qdrant (образ `qdrant/qdrant`); Redis (образ `redis`); PostgreSQL (образ `postgres`, минимум две реплики: primary и standby). Python-пакеты: `minio`/`boto3`, `qdrant-client`, `sentence-transformers` (модель эмбеддингов `all-MiniLM-L6-v2`, 384-мерные векторы, метрика Cosine), `redis`, `psycopg`.
- **GPU не требуется.** Эмбеддинги можно считать на CPU; при наличии inference-эндпоинта из ЛР2–ЛР3 допускается использовать его эндпоинт эмбеддингов.
- **Время.** 4 аудиторных часа + СРС на оформление отчёта.
- **Допустимые материалы.** Официальная документация MinIO/Qdrant/Redis/PostgreSQL, шаблоны из [`resources/`](../resources/README.md). Секреты (пароли, ключи доступа) не коммитить в репозиторий — использовать `.env`/K8s Secret.

## Задание

Задание строится по уровням КРМ — каждый следующий включает предыдущий.

- **Базовый (Б).** Развернуть три компонента через Docker Compose: MinIO, Qdrant, Redis. В MinIO создать bucket `corpus` и загрузить демо-корпус (`mc alias set local http://localhost:9000 <user> <pass>`, `mc mb local/corpus`, `mc cp ./corpus/*.txt local/corpus/`). В Qdrant создать коллекцию `docs` (размерность 384, distance Cosine). Написать скрипт ingestion: прочитать документы из MinIO, разбить на чанки, посчитать эмбеддинги (`sentence-transformers`), выполнить upsert точек в Qdrant с метаданными (`source`, `chunk_id`). Подтвердить наполнение: количество точек в коллекции равно числу чанков.

- **Средний (С).** Реализовать retrieval и кэширование. Скрипт retrieval: по текстовому запросу считает эмбеддинг, выполняет `search` top-k в Qdrant, собирает контекст, ключ запроса (hash) и результат кладёт в Redis с TTL (`SETEX`), при повторном запросе отдаёт из кэша. Развернуть PostgreSQL с потоковой репликацией primary → standby (`wal_level=replica`, `max_wal_senders`, запись в `pg_hba.conf`, `pg_basebackup -R` для standby); подтвердить репликацию через `SELECT * FROM pg_stat_replication` на primary и `SELECT pg_is_in_recovery()` (= `t`) на standby. Реализовать схему доступа RBAC: в PostgreSQL роль `rag_reader` (`GRANT SELECT`) и `rag_writer` (`GRANT SELECT, INSERT, UPDATE`); в MinIO политики `readonly` (для инференса) и `readwrite` (для ingestion) с привязкой к отдельным пользователям (`mc admin user add`, `mc admin policy attach`); в Qdrant read-only API-ключ для контура инференса. Показать, что read-only-учётка не может писать (ожидаемая ошибка прав).

- **Продвинутый (П).** Обосновать выбор стека: сравнительная таблица Qdrant vs Milvus vs Pinecone по критериям (тип индекса, фильтрация по метаданным, latency/throughput, self-hosted vs managed, сложность эксплуатации, стоимость) и аргументированный выбор под профиль «30 пользователей». Измерить retrieval-latency (p50/p95) с кэшем и без и показать эффект Redis. Продемонстрировать отказоустойчивость данных: сделать бэкап (`pg_basebackup`/snapshot MinIO через `mc mirror`), сымитировать отказ primary и переключить контур инференса на standby (promote), зафиксировать, что retrieval продолжает работать. Развернуть контур в Kubernetes (StatefulSet PostgreSQL + Qdrant + Redis, Secret для учёток, отдельные ServiceAccount/NetworkPolicy для read-only и read-write контуров) как продолжение ЛР3.

## Исходные данные и ресурсы

- **Демо-корпус.** 300–500 текстовых фрагментов (например, конспекты лекций дисциплины или публичный набор контекстов, см. [`resources/datasets/`](../resources/datasets/README.md)). Формат — `.txt`/`.md`, кодировка UTF-8.
- **Модель эмбеддингов.** `sentence-transformers/all-MiniLM-L6-v2` (384 dim) либо эндпоинт эмбеддингов из ЛР2–ЛР3.
- **Шаблоны.** `docker-compose.yml` стенда, `.env.example`, скрипты `ingest.py`/`retrieve.py`, шаблон отчёта — в [`resources/`](../resources/README.md).
- **Библиотеки.** См. [`resources/software/python-libs/`](../resources/software/python-libs/README.md).

## Формат сдачи

- **Репозиторий** с `docker-compose.yml` (или манифестами K8s), скриптами `ingest.py` и `retrieve.py`, конфигурацией репликации PostgreSQL, скриптом настройки RBAC (`grants.sql`, команды `mc admin`), файлом `.env.example` (без реальных секретов) и `README` с командами запуска.
- **Отчёт** (`REPORT.md`): схема контура (MinIO/Qdrant/Redis/PostgreSQL и потоки read-only/read-write), вывод `pg_stat_replication` и `pg_is_in_recovery()`, лог успешного ingestion (число точек), пример запроса retrieval с top-k и подтверждением попадания в кэш (cache miss → cache hit), демонстрация запрета записи read-only-учётке, для уровня П — таблица сравнения векторных БД и замеры latency.
- **Скринкаст** (2–4 мин) или последовательность скриншотов ключевых шагов.
- **Способ сдачи** — ссылка на репозиторий/PR; **срок** — по регламенту потока.

| 6 | С | Собран RAG-контур (MinIO + Qdrant + Redis), проходит ingestion и retrieval; настроена репликация PostgreSQL; реализована базовая схема доступа read-only/read-write хотя бы в PostgreSQL и MinIO. |

## Правила использования генеративного ИИ и внешних ресурсов

- Разрешено: официальная документация MinIO/Qdrant/Redis/PostgreSQL, использование БЯМ как помощника для генерации boilerplate (docker-compose, шаблоны скриптов) и разбора ошибок.
- Должно быть сделано самостоятельно и понято: проектирование схемы доступа read-only/read-write, конфигурация репликации PostgreSQL, обоснование выбора векторной БД. На защите обучающийся объясняет, почему выданы именно такие права и как проверена репликация.
- Если использование БЯМ является частью работы, зафиксируйте ключевые промпты и источники в приложении к отчёту. Запрещено коммитить реальные секреты и учётные данные.
