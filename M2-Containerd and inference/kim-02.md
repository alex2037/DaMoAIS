# КИМ модуля 2. Контейнеризация и серверы инференса моделей МО — ЛР2: Контейнеризация модели и запуск inference-сервера

## Назначение

КИМ проверяет способность обучающегося упаковать модель МО в переносимый контейнерный образ и запустить inference-endpoint на двух платформах — CPU и GPU — под SLA-контракт из ЛР1. Работа даёт первый работающий компонент сквозного проекта и первые эмпирические цифры latency/throughput, на которые опираются оркестрация (ЛР3), мониторинг (ЛР6) и A/B-сравнение (ЛР7).

## Проверяемые результаты обучения

| Индикатор КРМ | Уровень | Что проверяется |
| --- | --- | --- |
| LC-5.1 | Б–С | Сборка образа модели (многоэтапная, кэш слоёв, контроль размера), выбор и запуск сервера инференса (vLLM/Triton), понимание критериев сравнения серверов |
| BD-5.2 | С | Развёртывание одной модели на CPU (Docker) и на GPU (NVIDIA Container Toolkit), снятие базовых метрик latency/throughput, объяснение разницы CPU vs GPU |

## Условия выполнения

- Среда: Linux с установленным Docker (или containerd/Podman). Для GPU-части — хост с GPU NVIDIA, установленным драйвером и **NVIDIA Container Toolkit**; при отсутствии GPU допускается учебный GPU-узел курса или облачная GPU-VM.
- Инструменты: Docker, сервер инференса на выбор (**vLLM** для LLM или **Triton Inference Server**), утилиты замера (`curl`, `ab`, при желании k6), `nvidia-smi` для контроля GPU.
- Модель: небольшая открытая модель, помещающаяся в доступную память (например, LLM класса 0.5B–1B для GPU и уменьшенная/квантованная для CPU; либо классическая модель под Triton).
- Отводимое время: 4 ч аудиторно (ЛР) + 4 ч СРС (сравнение серверов инференса и обоснование выбора).
- Допустимые материалы: официальная документация Docker/NVIDIA Container Toolkit/vLLM/Triton, SLA-контракт из ЛР1.

## Задание

Задание строится по уровням КРМ — каждый следующий включает предыдущий.

- **Базовый (Б).** Написать **многоэтапный** `Dockerfile` для сервиса инференса: стадия сборки зависимостей и тонкая runtime-стадия; веса модели отделены от кода, настроен `.dockerignore`, порядок инструкций обеспечивает переиспользование кэша слоёв. Собрать образ и запустить inference-endpoint на CPU, убедиться, что endpoint отвечает.
  - сборка: `docker build -t infer:cpu .`
  - запуск (пример vLLM): `docker run --rm -p 8000:8000 infer:cpu --model <model> --device cpu`
  - проверка: `curl http://localhost:8000/v1/completions -d '{"model":"<model>","prompt":"ping","max_tokens":16}'`
  - размер образа: `docker images infer:cpu`
- **Средний (С).** Запустить тот же образ на GPU через NVIDIA Container Toolkit; убедиться, что GPU виден внутри контейнера (`nvidia-smi`). Снять **базовые метрики** на обеих платформах при одинаковом наборе запросов: latency (p50/p95), throughput (RPS и/или TPS токенов), для LLM — TTFT. Свести результаты CPU vs GPU в таблицу.
  - запуск на GPU: `docker run --rm --gpus all -p 8000:8000 infer:gpu --model <model>`
  - контроль GPU: `docker run --rm --gpus all infer:gpu nvidia-smi`
  - замер: `ab -n 500 -c 8 -p req.json -T application/json http://localhost:8000/v1/completions` (или сценарий k6)
- **Продвинутый (П).** Обосновать выбор сервера инференса под задачу «shared-сервис на 30 пользователей» по критериям (latency, throughput, GPU, batching, model parallelism, лицензирование, сложность эксплуатации). Продемонстрировать эффект оптимизаций: включить continuous batching и/или квантование (FP16→INT8) и показать изменение throughput/latency/GPU utilization. Оптимизировать размер образа (slim/distroless, отделение весов) и зафиксировать «до/после». Сопоставить полученные цифры с целевыми SLO из SLA-контракта ЛР1.

## Исходные данные и ресурсы

- SLA-контракт из ЛР1 (целевые SLO по latency/throughput).
- Инструменты и образы (Docker, NVIDIA Container Toolkit, vLLM, Triton) — [`resources/software/python-libs`](../resources/software/python-libs/README.md).
- Ориентиры latency/throughput CPU vs GPU — [`resources/benchmarks`](../resources/benchmarks/README.md).
- Набор запросов/промптов для замеров — [`resources/datasets`](../resources/datasets/README.md).
- Примеры `Dockerfile` и конфигураций серверов — [`resources/other`](../resources/other/README.md).
- Шаблон отчёта: `Dockerfile` и обоснование стадий → команды запуска CPU/GPU → таблица метрик CPU vs GPU → эффект оптимизаций → сопоставление с SLO ЛР1.

## Формат сдачи

- Репозиторий (или каталог проекта) с: `Dockerfile` (многоэтапный), `.dockerignore`, конфигурацией сервера инференса, скриптом/командами замеров.
- Собранный образ (тег и/или digest; при наличии — push в учебный registry).
- Отчёт (Markdown/PDF): команды сборки и запуска на CPU и GPU, таблица базовых метрик (latency p50/p95, throughput, TTFT), размер образа «до/после», сопоставление с SLO из ЛР1.
- Короткий скринкаст (по требованию преподавателя) с демонстрацией запуска на CPU и на GPU (`nvidia-smi` внутри контейнера).
- Способ и срок сдачи — согласно расписанию модуля.

## Критерии и шкала оценивания (10-балльная)

| Балл | Уровень | Критерий |
| --- | --- | --- |
| 10 | П | Многоэтапный оптимизированный образ; endpoint работает на CPU и GPU; сняты метрики на обеих платформах; продемонстрирован эффект continuous batching/квантования; выбор сервера обоснован по критериям; результаты сопоставлены со SLO ЛР1 |
| 8 | С–П | Образ собран многоэтапно, endpoint запущен на CPU и GPU, сняты базовые метрики и составлено сравнение CPU vs GPU; отдельные оптимизации или обоснование проработаны частично |
| 6 | С | Endpoint запущен на CPU и GPU, но замеры неполны либо сравнение поверхностно; образ собран без внимания к слоям/размеру |
| 4 | Б | Собран образ и запущен inference-endpoint на CPU, endpoint отвечает на запросы; GPU-часть или замеры отсутствуют |
| 0–3 | — | результат не достигнут: образ не собирается либо endpoint не запускается |

Порог сдачи ЛР — не ниже 4 баллов. Вес в итоговой оценке — см. [README, «Итоговая оценка»](../README.md#4-итоговая-оценка).

## Правила использования генеративного ИИ и внешних ресурсов

- Разрешено: официальная документация Docker/NVIDIA Container Toolkit/vLLM/Triton; использование БЯМ как помощника для черновика `Dockerfile` и разбора ошибок сборки/запуска.
- Должно быть сделано самостоятельно: постановка и проведение замеров, интерпретация разницы CPU vs GPU, обоснование выбора сервера инференса и сопоставление со SLO — это оценивается и не может быть заимствовано.
- Если `Dockerfile` или команды частично получены с помощью БЯМ, необходимо зафиксировать использованные промпты и источники в отчёте; оценивается понимание конфигурации и результатов, а не сгенерированный текст.
