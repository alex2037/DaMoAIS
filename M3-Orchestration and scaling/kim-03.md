# КИМ модуля 3. Оркестрация и масштабирование ИИ-сервисов — ЛР3: Развёртывание inference-сервиса в Kubernetes

## Назначение

КИМ проверяет способность обучающегося перевести контейнеризованный inference-сервис из ЛР2 в управляемое кластерное развёртывание в Kubernetes на GPU-ноде: с ресурсными лимитами, секретами, проверками живучести, автомасштабированием и обновлением без простоя. Работа реализует второй этап сквозного проекта («Оркестрация») и готовит сервис к подключению данных/RAG (ЛР4), CI/CD-поставке (ЛР5), мониторингу (ЛР6) и нагрузочному тестированию (ЛР7).

## Проверяемые результаты обучения

| Индикатор КРМ | Уровень | Что проверяется |
| --- | --- | --- |
| LC-5.1 | С | Развёртывание inference-сервиса в Kubernetes (Deployment/Service/Ingress), настройка `requests`/`limits`, secrets и probes, планирование на GPU-ноду (device plugin, node selectors, taints/tolerations) |
| LC-5.1 | С | Включение HPA (в т.ч. по кастомным/GPU-метрикам), демонстрация rolling update без простоя, обоснование выбора оркестратора под задачу |

## Условия выполнения

- Среда: кластер Kubernetes с доступной GPU-нодой. Допускается учебный кластер курса либо локальный кластер (kind/minikube/k3s); при отсутствии физического GPU GPU-планирование демонстрируется на учебном GPU-узле, а базовые части (requests/limits, probes, HPA по CPU) — локально.
- Компоненты: NVIDIA device plugin (ресурс `nvidia.com/gpu`), metrics-server для HPA; для GPU-метрик — DCGM-exporter + prometheus-adapter (при наличии стенда).
- Инструменты: `kubectl`, образ inference-сервиса из ЛР2 (в доступном registry), генератор нагрузки для проверки HPA (`ab`/k6/`hey`).
- Отводимое время: 4 ч аудиторно (ЛР) + 4 ч СРС (сравнение оркестраторов, проектирование манифестов).
- Допустимые материалы: официальная документация Kubernetes/NVIDIA GPU, SLA-контракт из ЛР1, образ и метрики из ЛР2.

## Задание

Задание строится по уровням КРМ — каждый следующий включает предыдущий.

- **Базовый (Б).** Развернуть inference-сервис из ЛР2 в Kubernetes: написать манифесты Deployment и Service; задать `requests`/`limits` по CPU/памяти; вынести токены/конфигурацию в Secret и ConfigMap; настроить liveness- и readiness-probe на HTTP-эндпоинт модели. Убедиться, что под в состоянии `Ready` и endpoint отвечает через Service.
  - применение: `kubectl apply -f deployment.yaml -f service.yaml`
  - статус: `kubectl get pods -o wide`, `kubectl describe pod <pod>`
  - проверка: `kubectl port-forward svc/infer 8000:8000` и `curl`
- **Средний (С).** Запланировать под на **GPU-ноду**: запросить ресурс `nvidia.com/gpu: 1`, добавить `nodeSelector`/`affinity` и `tolerations` под taint GPU-узла. Опубликовать сервис через Ingress. Включить **HPA** (минимум по CPU/памяти, при наличии стенда — по GPU-метрике/queue depth) и подтвердить масштабирование под нагрузкой. Проверить, что GPU виден в поде (`kubectl exec ... -- nvidia-smi`).
  - GPU в поде: `resources.limits: { nvidia.com/gpu: 1 }`
  - HPA: `kubectl autoscale deployment infer --min=1 --max=4 --cpu-percent=70` (или манифест HPA v2 с кастомной метрикой)
  - нагрузка для проверки: `ab -n 3000 -c 20 ...` и `kubectl get hpa -w`
- **Продвинутый (П).** Продемонстрировать **rolling update без простоя**: обновить образ/конфигурацию (`kubectl set image ...`), настроив `maxSurge`/`maxUnavailable` и корректный readiness-probe так, чтобы непрерывный поток запросов не получал ошибок во время выката; показать откат (`kubectl rollout undo`). Обосновать выбор Kubernetes против Docker Swarm/Nomad для задачи «shared-сервис на 30 пользователей на GPU» и связать настройки (реплики, лимиты, пороги HPA) с целевыми SLO из SLA-контракта ЛР1.
  - выкат и наблюдение: `kubectl set image deploy/infer infer=<new> && kubectl rollout status deploy/infer`
  - контроль отсутствия простоя: параллельный `ab`/k6 без ошибок во время выката

## Исходные данные и ресурсы

- Образ inference-сервиса и метрики из ЛР2; SLA-контракт из ЛР1.
- Инструменты (`kubectl`, kind/minikube/k3s, NVIDIA device plugin, DCGM-exporter, prometheus-adapter) — [`resources/software/python-libs`](../resources/software/python-libs/README.md).
- Примеры манифестов (Deployment, Service, Ingress, HPA, Secret) — [`resources/other`](../resources/other/README.md).
- Документация Kubernetes и NVIDIA GPU Operator — [`resources/textbooks`](../resources/textbooks/README.md).
- Ориентиры для requests/limits и порогов HPA — [`resources/benchmarks`](../resources/benchmarks/README.md).
- Шаблон отчёта: манифесты и обоснование ресурсов → размещение на GPU-ноде → настройка и проверка HPA → журнал rolling update (без ошибок) → обоснование выбора оркестратора и связь с SLO ЛР1.

## Формат сдачи

- Репозиторий (или каталог проекта) с манифестами Kubernetes: Deployment, Service, Ingress, Secret/ConfigMap, HPA (и, при наличии, конфигурация GPU-метрик).
- Отчёт (Markdown/PDF): вывод `kubectl get pods/hpa`, подтверждение размещения на GPU-ноде (`nvidia-smi` в поде), результат проверки HPA под нагрузкой, журнал rolling update с доказательством отсутствия простоя, обоснование выбора оркестратора.
- Короткий скринкаст (по требованию преподавателя): развёртывание, срабатывание HPA и rolling update без простоя.
- Способ и срок сдачи — согласно расписанию модуля; работа может выноситься на peer-review.

## Критерии и шкала оценивания (10-балльная)

| Балл | Уровень | Критерий |
| --- | --- | --- |
| 10 | П | Сервис развёрнут на GPU-ноде с requests/limits, secrets, probes; HPA работает (по GPU/кастомной метрике при наличии стенда); rolling update выполнен без простоя с откатом; выбор оркестратора обоснован, настройки связаны со SLO ЛР1 |
| 8 | С–П | Сервис на GPU-ноде через Ingress, HPA включён и подтверждён под нагрузкой, rolling update показан; отдельные элементы (GPU-метрика, доказательство отсутствия простоя) проработаны частично |
| 6 | С | Сервис развёрнут с requests/limits, secrets и probes, размещён на GPU-ноде; HPA настроен, но масштабирование не подтверждено нагрузкой |
| 4 | Б | Deployment и Service развёрнуты, заданы requests/limits и probes, endpoint отвечает; GPU-планирование или HPA отсутствуют |
| 0–3 | — | результат не достигнут: сервис не разворачивается или под не переходит в `Ready` |

Порог сдачи ЛР — не ниже 4 баллов. Вес в итоговой оценке — см. [README, «Итоговая оценка»](../README.md#4-итоговая-оценка).

## Правила использования генеративного ИИ и внешних ресурсов

- Разрешено: официальная документация Kubernetes/NVIDIA/HPA; использование БЯМ как помощника для черновиков манифестов и разбора ошибок планирования/probes.
- Должно быть сделано самостоятельно: настройка GPU-планирования и HPA, проведение и интерпретация rolling update без простоя, обоснование выбора оркестратора и связь настроек со SLO — это оценивается и не может быть заимствовано.
- Если манифесты частично получены с помощью БЯМ, необходимо зафиксировать использованные промпты и источники в отчёте; оценивается понимание конфигурации кластера и результатов, а не сгенерированный текст.
