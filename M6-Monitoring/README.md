# Модуль 6. Мониторинг, логирование и наблюдаемость ИИ-систем

## Место в дисциплине и связь со сквозным проектом

Модуль соответствует разделу 6 РПД и лекции 8. К этому моменту сквозной проект развёрнут в Kubernetes (ЛР3), обеспечен данными и RAG-контуром (ЛР4) и поставляется через CI/CD + GitOps (ЛР5). Модуль делает сервис наблюдаемым: добавляет экспорт метрик инференса в Prometheus, дашборды и алерты в Grafana, сбор логов в Loki/ELK и корреляцию логов с метриками. Наблюдаемость из ЛР6 — обязательное условие следующего этапа сквозного проекта: без метрик TTFT/TPS/GPU util/queue depth невозможно провести осмысленное нагрузочное тестирование (ЛР7) и доказать SLA в итоговом проекте (ЛР8).

Проверяемые индикаторы КРМ (версия 3.0): **LC-5.1** (инструменты мониторинга и практики сопровождения) и **BD-5.3** (измерение метрик производительности и оценка качества). Целевой уровень модуля — **средний (С)**.

## Содержание лекции

- **Стек наблюдаемости** — Prometheus (модель pull/scrape, PromQL, exporters), Grafana (дашборды, панели, переменные, datasource), логи в Loki (+ Promtail) или ELK (Filebeat → Elasticsearch → Kibana); три столпа observability: метрики, логи, трейсы.
- **Ключевые метрики инференса** — TTFT (time to first token), inter-token latency, TPS (tokens per second), throughput (req/s), GPU utilization, queue depth (ожидающие запросы), латентность p50/p95/p99; какие из них SLO-метрики.
- **Инструментирование** — экспорт метрик из сервиса (Prometheus client / встроенный эндпоинт `/metrics`, у vLLM — `vllm:*` метрики); GPU-метрики через NVIDIA DCGM exporter (`DCGM_FI_DEV_GPU_UTIL`).
- **Логирование и трейсинг** — структурированные (JSON) логи, correlation-id/`trace_id` на запрос; распределённый трейсинг (OpenTelemetry) для разбиения latency по этапам (retrieval → генерация).
- **Дашборды и алерты** — панели PromQL (`histogram_quantile`, `rate`); Alertmanager, правила на основе SLO (высокий p95 TTFT, рост queue depth, насыщение GPU), маршрутизация нотификаций.
- **Корреляция логов и метрик** — общие метки (labels) и `trace_id`; переход из панели Grafana в связанные логи Loki (exemplars/split-view); диагностика инцидента «метрика → лог → причина».

## Экспресс-тест текущего контроля

**Вопрос 1.** Что измеряет метрика TTFT (time to first token) для LLM-инференса?
- A) суммарное время генерации всего ответа
- B) задержку до появления первого сгенерированного токена
- C) число токенов в секунду
- D) загрузку GPU в процентах

*Верный ответ: B.*

**Вопрос 2.** По какой модели Prometheus собирает метрики с сервисов?
- A) push: сервис сам отправляет метрики в Prometheus
- B) pull/scrape: Prometheus периодически опрашивает эндпоинт `/metrics`
- C) через очередь сообщений Kafka
- D) только по SNMP

*Верный ответ: B.*

**Вопрос 3.** Какая PromQL-конструкция даёт p95 латентности из гистограммы?
- A) `avg(request_latency_seconds)`
- B) `sum(rate(request_latency_seconds_count[5m]))`
- C) `histogram_quantile(0.95, rate(request_latency_seconds_bucket[5m]))`
- D) `max(request_latency_seconds)`

*Верный ответ: C.*

**Вопрос 4.** Метрика queue depth (число ожидающих запросов) в контексте инференса сигнализирует прежде всего о:
- A) объёме занятого дискового пространства
- B) насыщении сервиса: запросы прибывают быстрее, чем обрабатываются
- C) количестве строк в логах
- D) версии модели

*Верный ответ: B.*

**Вопрос 5.** Что технически позволяет связать конкретную запись в логах (Loki/ELK) с всплеском на графике метрик в Grafana?
- A) одинаковый шрифт панелей
- B) общие метки/`trace_id` и единая временная ось (exemplars, split-view)
- C) ручной поиск по времени в двух вкладках без общих идентификаторов
- D) экспорт логов в Excel

*Верный ответ: B.*
