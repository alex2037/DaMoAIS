# КИМ модуля 6. Мониторинг, логирование и наблюдаемость ИИ-систем — ЛР6: Мониторинг и наблюдаемость ИИ-сервиса

## Назначение

КИМ проверяет способность обучающегося сделать развёрнутый inference-сервис наблюдаемым: инструментировать его для экспорта метрик, собрать метрики в Prometheus, построить дашборд ключевых показателей инференса и настроить алерты в Grafana, собрать логи в Loki/ELK и связать логи с метриками. Наблюдаемость — обязательный вход в ЛР7 (нагрузочное тестирование) и ЛР8 (доказательство SLA): без метрик TTFT/TPS/GPU util/queue depth нельзя измеримо оценить эксплуатационное качество сквозного проекта.

## Проверяемые результаты обучения

| Индикатор КРМ | Уровень | Что проверяется |
| --- | --- | --- |
| LC-5.1 | С | Разворачивает Prometheus + Grafana + Loki/ELK, инструментирует сервис (эндпоинт `/metrics`, DCGM exporter для GPU), настраивает scrape и правила Alertmanager. |
| BD-5.3 | С | Измеряет метрики инференса (TTFT, TPS, GPU utilization, queue depth, p50/p95/p99) через Prometheus/Grafana и строит дашборд, отражающий состояние сервиса под нагрузкой. |

## Условия выполнения

- **Среда.** Kubernetes из ЛР3 (`kind`/`minikube` допустимы) либо Docker Compose-стенд. `kubectl`, `helm` (опционально для установки kube-prometheus-stack).
- **Инструменты (реальные, без выдуманных версий).** Prometheus, Grafana, Alertmanager; Loki + Promtail **или** ELK (Elasticsearch + Filebeat + Kibana); при наличии GPU — NVIDIA DCGM exporter. Наблюдаемый сервис — inference-эндпоинт из ЛР2–ЛР3 (vLLM отдаёт метрики `vllm:*` на `/metrics`) либо FastAPI-сервис, инструментированный `prometheus-client`.
- **Генератор нагрузки.** `k6`, `locust` или `ab` — чтобы метрики и дашборд наполнились данными (полноценный нагрузочный тест — предмет ЛР7).
- **GPU не обязателен.** Без GPU панель GPU utilization помечается как «н/д»; остальные метрики (TTFT/TPS/queue depth/latency) обязательны.
- **Время.** 4 аудиторных часа + СРС.
- **Ограничения.** Конфигурации Prometheus/Grafana/Alertmanager и дашборд (JSON) хранить в репозитории как код; секреты нотификаций — вне репозитория.

## Задание

Задание строится по уровням КРМ — каждый следующий включает предыдущий.

- **Базовый (Б).** Развернуть Prometheus и Grafana. Инструментировать inference-сервис: включить эндпоинт `/metrics` (у vLLM он встроен; для FastAPI подключить `prometheus-client` — гистограмма латентности, счётчики запросов/токенов). Настроить scrape сервиса в Prometheus и подключить Prometheus как datasource в Grafana. Подтвердить, что метрики видны в Prometheus (`up == 1`, целевые метрики присутствуют).

- **Средний (С).** Построить дашборд Grafana с панелями: TTFT (`histogram_quantile(0.95, rate(vllm:time_to_first_token_seconds_bucket[5m]))` или аналог для FastAPI), TPS (`rate(vllm:generation_tokens_total[1m])`), queue depth (`vllm:num_requests_waiting`), p50/p95/p99 e2e-latency, throughput (req/s), при наличии GPU — GPU utilization (`DCGM_FI_DEV_GPU_UTIL`). Настроить в Alertmanager минимум два правила по SLO (например: p95 TTFT выше порога дольше N минут; queue depth растёт). Собрать логи сервиса в Loki (+ Promtail) или ELK и подключить как datasource в Grafana. Прогнать генератор нагрузки, чтобы дашборд и алерты наполнились реальными данными; показать сработавший алерт.

- **Продвинутый (П).** Реализовать корреляцию логов и метрик: добавить `trace_id`/correlation-id в структурированные (JSON) логи и в метки, обеспечить переход из панели Grafana в связанные логи (exemplars или split-view Loki). Подключить распределённый трейсинг (OpenTelemetry) и разложить latency по этапам (retrieval из ЛР4 → генерация). Оформить дашборд как SLO-панель с error budget и обосновать пороги алертов относительно SLA-контракта из ЛР1. Провести разбор одного инцидента по цепочке «метрика (всплеск p95) → лог (`trace_id`) → причина».

## Исходные данные и ресурсы

- **Наблюдаемый сервис.** Inference-эндпоинт из ЛР2–ЛР3 (vLLM/Triton) или FastAPI-сервис с `prometheus-client`.
- **Шаблоны.** Конфиг scrape Prometheus, правила Alertmanager, дашборд Grafana (JSON), конфиг Promtail/Filebeat, скрипт генерации нагрузки — в [`resources/`](../resources/README.md) и [`resources/other/`](../resources/other/README.md).
- **Ориентиры значений метрик.** См. [`resources/benchmarks/`](../resources/benchmarks/README.md).
- **Библиотеки/ПО.** См. [`resources/software/python-libs/`](../resources/software/python-libs/README.md).

## Формат сдачи

- **Репозиторий** с конфигурациями Prometheus (scrape), Alertmanager (правила), дашбордом Grafana (экспорт JSON), конфигом сбора логов (Promtail/Filebeat), кодом инструментирования сервиса и `README` с командами запуска стенда.
- **Отчёт** (`REPORT.md`): архитектура стека наблюдаемости; скриншот дашборда с TTFT, TPS, queue depth, latency (и GPU util при наличии) под нагрузкой; используемые PromQL-выражения; скриншот сработавшего алерта; демонстрация связки лог ↔ метрика (для уровня П — разбор инцидента по `trace_id`).
- **Скринкаст** (2–4 мин): наполнение дашборда под нагрузкой и срабатывание алерта.
- **Способ сдачи** — ссылка на репозиторий/PR; **срок** — по регламенту потока.

| 6 | С | Построен дашборд Grafana с ключевыми метриками инференса; настроен хотя бы один рабочий алерт; логи собираются в Loki/ELK. |

## Правила использования генеративного ИИ и внешних ресурсов

- Разрешено: официальная документация Prometheus/Grafana/Loki/ELK/DCGM; использование БЯМ как помощника для генерации конфигов scrape, правил Alertmanager, PromQL-выражений и JSON дашборда.
- Должно быть сделано самостоятельно и понято: выбор и смысл метрик инференса (что означает TTFT/TPS/queue depth), составление PromQL, обоснование порогов алертов относительно SLO. На защите обучающийся объясняет, о чём сигнализирует каждая панель и почему выбраны такие пороги.
- При использовании БЯМ как части работы зафиксируйте ключевые промпты и источники в приложении к отчёту. Запрещено коммитить секреты каналов нотификаций.
