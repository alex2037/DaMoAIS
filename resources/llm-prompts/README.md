# LLM-сервисы и промпты

Промпты для больших языковых моделей (БЯМ), применяемые как вспомогательный инструмент при выполнении лабораторных и подготовке материалов. Модель обучает студентов работать с ИИ осознанно: результат промпта всегда проверяется и адаптируется, а не копируется как есть (см. правила в [методических указаниях обучающимся](../../methodical-guidelines/students/README.md)).

| Назначение | Связанные КИМ | Ожидаемый результат | Способ верификации |
|---|---|---|---|
| Генерация многоэтапного Dockerfile для inference-сервиса | [Модуль 2](<../../M2-Containerd and inference/README.md>) | Черновик Dockerfile | Сборка образа и запуск локально |
| Генерация манифестов Kubernetes (Deployment/Service/HPA) | [Модуль 3](<../../M3-Orchestration and scaling/README.md>) | YAML-манифесты | `kubectl apply --dry-run=server` |
| Генерация нагрузочного сценария k6 | [Модуль 7](<../../M7-Load testing and quality assessment/README.md>) | JS-скрипт k6 | Тестовый прогон `k6 run` |
| Разбор метрик и формулировка алертов PromQL | [Модуль 6](<../../M6-Monitoring/README.md>) | Выражения PromQL + пороги | Проверка на реальных рядах в Grafana |
| Черновик SLA-контракта по описанию сервиса | [Модуль 1](<../../M1-Hardware architecture of AI/README.md>) | Таблица SLO | Ревью на реалистичность целевых значений |

## Примеры промптов

Промпты сформулированы под любую БЯМ (self-hosted vLLM, публичные сервисы). Плейсхолдеры в угловых скобках заменяются под задачу.

**1. Dockerfile для inference-сервиса (Модуль 2).**

> Ты DevOps-инженер. Составь многоэтапный Dockerfile для сервиса инференса на базе <vLLM/Triton> для модели <название модели>. Первый этап — сборка зависимостей, второй — минимальный runtime-образ. Учти проброс GPU (CUDA), непривилегированного пользователя и кэш слоёв. Дай только Dockerfile и краткое пояснение выбора базового образа.

**2. Манифесты Kubernetes с HPA (Модуль 3).**

> Сгенерируй манифесты Kubernetes для развёртывания inference-сервиса из образа <образ:тег> на GPU-ноде: Deployment (requests/limits, `nvidia.com/gpu: 1`, liveness/readiness probes), Service (ClusterIP) и HPA по метрике <метрика>. Добавь tolerations для taint <ключ>. Верни валидный YAML.

**3. Нагрузочный сценарий k6 «30 пользователей» (Модуль 7).**

> Напиши сценарий k6 на JavaScript для нагрузочного теста HTTP-эндпоинта <URL> с профилем 30 одновременных пользователей: ramp-up 1 мин, плато 5 мин, ramp-down 1 мин. Тело запроса — JSON вида <пример>. Добавь пороги (thresholds) по p95 < <N> мс и доле ошибок < 1%.

**4. Разбор метрик и алерты (Модуль 6).**

> Дан набор метрик инференса: TTFT, TPS, `vllm:num_requests_waiting`, `DCGM_FI_DEV_GPU_UTIL`. Предложи 5 правил алертов PromQL с порогами и пояснениями, привязанных к SLO <перечень SLO>. Укажи окна усреднения и как избежать ложных срабатываний.

**5. Черновик SLA-контракта (Модуль 1).**

> Составь SLA-контракт для shared ИИ-сервиса на 30 пользователей в формате таблицы: показатель (p95 latency, TTFT, throughput, availability), целевое значение, способ измерения, последствия нарушения. Значения предложи реалистичные для self-hosted LLM-сервиса и поясни, откуда они.
