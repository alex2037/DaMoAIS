# КИМ модуля 5. CI/CD, реестры моделей и GitOps — ЛР5: Автоматизация поставки модели

## Назначение

КИМ проверяет способность обучающегося автоматизировать поставку ИИ-сервиса: собрать CI-пайплайн, который по коммиту собирает контейнерный образ модели, публикует его в реестр и через GitOps (ArgoCD) декларативно разворачивает в Kubernetes. Отдельно проверяется управляемость поставки: автоматический деплой по изменению в git и откат через `git revert`. Работа завершает этап 3 сквозного проекта («Данные/RAG и поставка») и делает inference-сервис из ЛР2–ЛР4 воспроизводимо поставляемым.

## Проверяемые результаты обучения

| Индикатор КРМ | Уровень | Что проверяется |
| --- | --- | --- |
| LC-5.1 | С | Настраивает CI (GitHub Actions или GitLab CI): сборка образа модели, тесты, push в OCI-реестр; настраивает ArgoCD Application с автосинхронизацией на репозиторий манифестов. |
| LC-5.1 | С | Демонстрирует end-to-end поставку по коммиту (обновление тега образа → sync ArgoCD → новый pod) и откат через `git revert` (возврат к предыдущему образу без ручных действий в кластере). |

## Условия выполнения

- **Среда.** Локальный Kubernetes — `kind` или `minikube`; `kubectl`, `kustomize`. Аккаунт GitHub (для GitHub Actions + GHCR) или GitLab (GitLab CI + Container Registry). Docker для сборки.
- **Инструменты (реальные, без выдуманных версий).** GitHub Actions или GitLab CI; OCI-реестр (GHCR / GitLab Container Registry); ArgoCD (манифест установки `install.yaml` из upstream). Опционально MLflow Model Registry для версионирования модели.
- **Приложение для деплоя.** Inference-сервис из ЛР2–ЛР3 (или упрощённый FastAPI-эндпоинт инференса), упакованный в образ.
- **GPU не требуется** — допускается CPU-образ; фокус работы на автоматизации поставки, а не на производительности.
- **Время.** 3 аудиторных часа + СРС.
- **Ограничения.** Секреты (токен реестра, kubeconfig) — только через GitHub/GitLab secrets или OIDC; не коммитить в репозиторий. Тег образа — по git-SHA (immutable), а не `latest`.

## Задание

Задание строится по уровням КРМ — каждый следующий включает предыдущий.

- **Базовый (Б).** Написать CI-workflow (`.github/workflows/build.yml` или `.gitlab-ci.yml`), который по push в основную ветку собирает образ inference-сервиса (`docker build`), тегирует его git-SHA и публикует в реестр (`docker push ghcr.io/<user>/<app>:<sha>`). Аутентификация в реестр — через secret/OIDC. Подтвердить, что образ появился в реестре.

- **Средний (С).** Развернуть ArgoCD в кластере (`kubectl create namespace argocd && kubectl apply -n argocd -f <upstream install.yaml>`). Подготовить репозиторий (или каталог) манифестов с `kustomization.yaml` и Deployment/Service inference-сервиса. Создать `Application` ArgoCD, указывающий на этот репозиторий, с политикой `automated` (`selfHeal: true`, `prune: true`). Добавить в CI шаг, который после push образа обновляет тег в манифестах (`kustomize edit set image app=ghcr.io/<user>/<app>:<sha>`) и коммитит в репозиторий манифестов. Продемонстрировать сквозной цикл: коммит в код → сборка → push → обновление манифеста → автоматический sync ArgoCD → новый pod с новым образом. Добавить в CI smoke-тест эндпоинта инференса перед публикацией.

- **Продвинутый (П).** Реализовать управляемый откат: внести заведомо ломающее изменение, увидеть неуспешный деплой, выполнить `git revert` коммита манифеста и показать, что ArgoCD автоматически вернул предыдущий рабочий образ (кластер не трогали руками). Разделить репозиторий приложения и репозиторий манифестов (app-of-apps или отдельный manifests-repo). Подключить MLflow Model Registry: промоушен версии модели Staging → Production триггерит обновление образа. Составить сравнительную таблицу GitHub Actions / GitLab CI / ArgoCD / Tekton по критериям (модель исполнения, K8s-native, secrets, кривая обучения) и обосновать выбранный стек для профиля «30 пользователей».

## Исходные данные и ресурсы

- **Приложение.** Inference-сервис из ЛР2–ЛР3 или минимальный FastAPI-эндпоинт `/predict`.
- **Шаблоны.** `Dockerfile` (multi-stage), заготовка CI-workflow, `kustomization.yaml`, `Application`-манифест ArgoCD, `README` со сценарием запуска — в [`resources/`](../resources/README.md) и [`resources/problem-banks/`](../resources/problem-banks/README.md).
- **Библиотеки/ПО.** См. [`resources/software/python-libs/`](../resources/software/python-libs/README.md).

## Формат сдачи

- **Репозиторий (репозитории)** с CI-workflow, `Dockerfile`, манифестами (`kustomization.yaml`, Deployment/Service), `Application`-манифестом ArgoCD.
- **Отчёт** (`REPORT.md`): схема пайплайна «commit → build → registry → ArgoCD → K8s»; ссылка на успешный прогон CI; скриншот образа в реестре; скриншот ArgoCD (статус Synced/Healthy); демонстрация деплоя по коммиту и отката через `git revert` (лог/скриншоты «до» и «после»); для уровня П — таблица сравнения платформ и описание промоушена модели.
- **Скринкаст** (2–4 мин) сквозного цикла деплоя и отката.
- **Способ сдачи** — ссылка на репозиторий/PR; **срок** — по регламенту потока.

## Критерии и шкала оценивания (10-балльная)

| Балл | Уровень | Критерий |
| --- | --- | --- |
| 10 | П | Полный GitOps-контур с разделением repo приложения/манифестов; продемонстрирован автоматический откат через `git revert`; интеграция MLflow-промоушена; обоснованное сравнение CI/CD-платформ. |
| 8 | С–П | Сквозной цикл commit → build → registry → sync ArgoCD → новый pod работает; есть smoke-тест в CI; откат через `git revert` показан; часть продвинутого уровня выполнена. |
| 6 | С | CI собирает и публикует образ; ArgoCD развёрнут и синхронизирует Application; обновление тега образа приводит к деплою в кластер. |
| 4 | Б | CI собирает образ и публикует в реестр по git-SHA; деплой в кластер выполняется, но без GitOps-автоматизации (ручной `kubectl`/sync). |
| 0–3 | — | Пайплайн не собирается, образ не публикуется, деплой не воспроизводится. |

Порог сдачи ЛР — не ниже 4 баллов. Вес в итоговой оценке — см. [README, «Итоговая оценка»](../README.md#4-итоговая-оценка).

## Правила использования генеративного ИИ и внешних ресурсов

- Разрешено: официальная документация GitHub Actions/GitLab CI/ArgoCD/Kustomize; использование БЯМ как помощника для генерации YAML-заготовок workflow и манифестов, разбора ошибок sync.
- Должно быть сделано самостоятельно и понято: устройство GitOps-цикла, связь CI и ArgoCD, механика отката через `git revert`, работа с secrets/OIDC. На защите обучающийся объясняет, что произойдёт в кластере при конкретном коммите и как откатиться.
- При использовании БЯМ как части работы зафиксируйте ключевые промпты и источники в приложении к отчёту. Запрещено коммитить токены реестра, kubeconfig и иные секреты.
