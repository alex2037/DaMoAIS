# Модуль 5. CI/CD, реестры моделей и GitOps

## Место в дисциплине и связь со сквозным проектом

Модуль соответствует разделу 5 РПД и лекции 7. К этому моменту сквозной проект имеет inference-сервис в Kubernetes (ЛР2–ЛР3) и слой данных с RAG-контуром (ЛР4). Модуль автоматизирует поставку: вместо ручного `docker build` и `kubectl apply` строится пайплайн «сборка образа модели → push в registry → GitOps-деплой в Kubernetes», в котором желаемое состояние декларативно хранится в git, а его применение в кластер выполняет ArgoCD. Результат ЛР5 завершает этап 3 сквозного проекта («Данные/RAG и поставка») и подготавливает почву для наблюдаемости (ЛР6) и нагрузочного тестирования (ЛР7).

Проверяемый индикатор КРМ (версия 3.0): **LC-5.1** (выбор инструментов и инженерных практик промышленной разработки, развёртывания и сопровождения моделей МО). Целевой уровень модуля — **средний (С)**.

## Результаты обучения

| Индикатор КРМ | Уровень | Что проверяется |
| --- | --- | --- |
| LC-5.1 | С | Настраивает CI (GitHub Actions или GitLab CI) для ML-пайплайна: сборка контейнерного образа модели, тесты, публикация в registry; связывает CI с GitOps-деплоем (ArgoCD) в Kubernetes. |
| LC-5.1 | С | Демонстрирует автоматический деплой по коммиту (изменение тега образа в git → синхронизация ArgoCD) и откат изменения через `git revert` без ручного вмешательства в кластер. |

## Содержание лекции

- **Сравнение CI/CD-платформ** — GitLab CI, GitHub Actions, ArgoCD, Tekton по критериям: модель исполнения (runners против pod-native), декларативность, интеграция с Kubernetes, secrets, экосистема; когда CI (сборка/тесты) и когда CD/GitOps (доставка в кластер).
- **ML-пайплайн** — сборка образа модели (Dockerfile, multi-stage, кэш слоёв); тесты (smoke-тест инференса, проверка контракта API, линт/валидация манифестов, скан образа); публикация в реестр.
- **Реестр моделей и реестр образов** — MLflow Model Registry (стадии Staging/Production, промоушен версий) против OCI-реестра образов (GHCR, GitLab Container Registry, Harbor); что версионируется и как связывается с деплоем.
- **GitOps-подход** — единый источник истины в git; ArgoCD Application; sync, self-heal, drift detection; pull-модель доставки против push; разделение репозитория приложения и репозитория манифестов.
- **Автоматизация обновления образа** — обновление тега через `kustomize edit set image` или Helm values; commit в манифесты → ArgoCD подхватывает изменение.
- **Откат** — `git revert` как основной способ отката в GitOps; история git = история состояния кластера; отличие от `kubectl rollout undo`.
- **Секреты в пайплайне** — GitHub Actions secrets / OIDC вместо долгоживущих ключей; sealed-secrets/SOPS для секретов в GitOps-репозитории.
- **Связь с MLOps** — continuous training, промоушен модели по стадиям, автоматическая поставка новой версии модели через тот же пайплайн.

## Экспресс-тест текущего контроля

**Вопрос 1.** В чём ключевое отличие GitOps-подхода (ArgoCD) от императивного `kubectl apply`?
- A) GitOps работает только с Helm, а `kubectl` — только с YAML
- B) желаемое состояние декларативно хранится в git и автоматически синхронизируется в кластер (pull-модель)
- C) GitOps не требует Kubernetes
- D) `kubectl apply` быстрее и потому предпочтителен в проде

*Верный ответ: B.*

**Вопрос 2.** Что версионирует OCI-реестр образов (GHCR/GitLab Registry), а что — MLflow Model Registry?
- A) оба версионируют только исходный код
- B) реестр образов — контейнерные образы, MLflow — версии/стадии моделей
- C) реестр образов — датасеты, MLflow — образы
- D) это одно и то же, названия взаимозаменяемы

*Верный ответ: B.*

**Вопрос 3.** Как в GitOps-контуре корректно откатить неудачный деплой?
- A) вручную удалить pod в кластере
- B) отключить ArgoCD, чтобы он не мешал
- C) выполнить `git revert` коммита с изменением образа — ArgoCD синхронизирует предыдущее состояние
- D) пересобрать образ с тем же тегом

*Верный ответ: C.*

**Вопрос 4.** Какая функция ArgoCD автоматически возвращает кластер к состоянию из git при ручном изменении ресурса?
- A) self-heal (автоисправление drift)
- B) horizontal pod autoscaling
- C) canary-анализ
- D) image pull policy

*Верный ответ: A.*

**Вопрос 5.** Почему тег образа лучше формировать по git-SHA коммита, а не использовать `latest`?
- A) `latest` не поддерживается в Kubernetes
- B) SHA даёт неизменяемую (immutable) привязку деплоя к конкретной сборке и воспроизводимость/откат
- C) SHA-теги занимают меньше места в реестре
- D) разницы нет

*Верный ответ: B.*
