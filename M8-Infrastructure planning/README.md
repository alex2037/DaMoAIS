# Модуль 8. Планирование мощности, разнородная инфраструктура и итоговый проект

## Место в дисциплине и связь со сквозным проектом

Модуль соответствует Разделу 8 РПД и завершает дисциплину. Он собирает результаты всех предыдущих этапов в единый shared ИИ-сервис на 30 человек: inference-endpoint из ЛР2–ЛР3, RAG-контур из ЛР4, поставку из ЛР5, мониторинг из ЛР6 и нагрузочные измерения из ЛР7. На основе цифр ЛР7 выполняется capacity planning (сколько ресурсов нужно на 30 пользователей и как это масштабируется до 1000), обосновывается выбор разнородной инфраструктуры (CPU/GPU/TPU, облако/on-prem/edge) и доказывается SLA-контракт, зафиксированный в ЛР1.

Модуль замыкает сквозную линию курса «запустил локально — работает» → «запустил в проде — выдерживает нагрузку 30 человек». Итоговая лабораторная ЛР8 является технической основой командного проекта; **командная защита проекта оценивается по отдельному интегральному КИМ — см. [Project/README.md](../Project/README.md)**.

Индикаторы: **BD-5.2, BD-5.3**. Уровень: **П**.

## Содержание лекции

- **Capacity planning:** расчёт требуемых ресурсов по результатам нагрузочных тестов ЛР7 — от измеренного throughput на реплику/GPU к числу GPU для 30 пользователей; экстраполяция на 1000 пользователей с учётом резерва (headroom), пиков и целевого p95; линейность и её пределы (насыщение, батчинг).
- **Разнородные ускорители:** CPU (универсальность, дешевизна, низкий throughput на LLM), GPU (высокий параллелизм, батчинг, стоимость), TPU (матричные операции, специфика фреймворков и доступности); когда какой уместен.
- **Облачная vs on-prem инфраструктура:** капитальные и операционные затраты, эластичность и время поставки ресурсов, приватность данных компании, привязка к поставщику; критерии выбора под задачу «30 пользователей».
- **Edge-развёртывание:** ONNX Runtime, TensorFlow Lite, целевые устройства (в том числе Raspberry Pi); оптимизация модели (квантование, компиляция) под ограниченные ресурсы; сценарии, где edge оправдан.
- **Гибридная схема:** on-prem GPU для базовой нагрузки + облачный fallback для пиков/аварий (burst); маршрутизация и квоты; экономический и эксплуатационный смысл гибрида.
- **Формулирование и доказательство SLA:** перевод результатов ЛР7 в проверяемые SLO (p95, availability, throughput); проверка SLA-контракта из ЛР1 данными; связь SLA с capacity planning и с итоговым проектом.

## Экспресс-тест текущего контроля

1. По ЛР7 одна GPU-реплика устойчиво держит 8 RPS при `p95 < 200 мс`. Профиль «30 пользователей» требует ~20 RPS в пике. Сколько GPU-реплик заложить с 25 % резервом?
   - а) 2; б) 3; в) 4; г) 8.

   **Верный ответ: б.** 20 / 8 = 2,5 → округляем вверх до 3; резерв 25 % подтверждает 3 реплики (3 × 8 = 24 RPS ≥ 25 RPS с запасом). Расчёт опирается на измеренный, а не заявленный throughput.

2. Какой runtime предназначен для инференса на устройствах с ограниченными ресурсами (мобильные, микроконтроллеры, Raspberry Pi)?
   - а) vLLM; б) TensorFlow Lite / ONNX Runtime; в) Triton на GPU-кластере; г) Spark.

   **Верный ответ: б.** TF Lite и ONNX Runtime рассчитаны на edge; vLLM/Triton — серверный GPU-инференс, Spark — обработка больших данных.

3. Зачем в гибридной схеме нужен облачный fallback при наличии on-prem GPU?
   - а) чтобы удешевить постоянную нагрузку;
   - б) чтобы обрабатывать пики и аварийные ситуации сверх ёмкости on-prem без нарушения SLA;
   - в) чтобы отказаться от мониторинга;
   - г) чтобы уменьшить размер модели.

   **Верный ответ: б.** On-prem GPU покрывает базовую нагрузку, облако подключается при всплесках/отказах (burst), сохраняя availability из SLA.

4. Что значит «доказать SLA» для итогового сервиса?
   - а) записать целевые значения в документ;
   - б) подтвердить целевые SLO (p95, availability, throughput) измерениями из нагрузочных и chaos-тестов ЛР7;
   - в) получить устное согласие заказчика;
   - г) увеличить число реплик до максимума.

   **Верный ответ: б.** SLA доказывается данными: SLO из ЛР1 сопоставляются с измеренными в ЛР7 метриками; без цифр это лишь декларация.
