# Модуль 7. Нагрузочное тестирование, устойчивость и оценка качества

## Место в дисциплине и связь со сквозным проектом

Модуль соответствует Разделу 7 РПД и относится к четвёртому этапу сквозного проекта — «Эксплуатация и защита» (ЛР6 → ЛР7 → ЛР8). К этому моменту сервис уже развёрнут в Kubernetes (ЛР3), обеспечен данными и RAG-контуром (ЛР4), поставляется через CI/CD (ЛР5) и инструментирован метриками в Prometheus/Grafana (ЛР6). Модуль отвечает на главный вопрос дисциплины: выдерживает ли сервис нагрузку группы из 30 пользователей и соответствует ли он SLA-контракту, зафиксированному в ЛР1.

Практический результат модуля — измеренные под нагрузкой метрики (p50/p95/p99, throughput, error rate, cost per 1K requests, W/req, GPU utilization), сравнение двух конфигураций (A/B) и подтверждённая chaos-проверками устойчивость. Эти данные — прямой вход в capacity planning и SLA-отчёт итогового проекта (ЛР8, Модуль 8): без цифр из ЛР7 нельзя ни доказать SLA, ни рассчитать требуемое число GPU.

Индикатор: **BD-5.3**. Уровень: **С–П**.

## Содержание лекции

- **Виды нагрузочных тестов:** soak (длительная стабильность, поиск утечек и деградации), spike (резкий скачок нагрузки и скорость восстановления), stress (нагрузка выше проектной для поиска точки отказа); отличие от baseline/smoke-теста.
- **Инструменты нагрузки:** k6 (сценарии на JavaScript, thresholds и checks, профили ramp-up/steady/ramp-down); Locust (сценарии на Python, распределённый режим master/worker, веб-интерфейс); Apache Bench `ab` (быстрый одиночный замер, ограничения). Критерии выбора инструмента под задачу.
- **Метрики производительности и эффективности:** латентность p50/p95/p99, throughput (RPS/TPS), error rate; для LLM — TTFT и inter-token latency; экономические и физические метрики — cost per 1K requests, энергопотребление W/req, GPU utilization %. Разница между средним и перцентилями, почему SLO задают по p95/p99.
- **A/B-тестирование конфигураций:** сравнение двух вариантов развёртывания при одинаковом профиле нагрузки (например, CPU vs GPU; разный размер батча/continuous batching; квантование FP16 vs INT8); интерпретация trade-off «латентность — стоимость — качество».
- **Chaos engineering:** гипотеза устойчивости и «взрывной радиус»; отказ пода (`kubectl delete pod`, Chaos Mesh PodChaos); сетевое разделение и задержки (Chaos Mesh NetworkChaos); наблюдение за восстановлением через probes, HPA и метрики; связь с availability из SLA.

## Экспресс-тест текущего контроля

1. Что означает метрика p95 latency = 180 мс?
   - а) средняя задержка запроса равна 180 мс;
   - б) 95 % запросов обработаны быстрее 180 мс, 5 % — медленнее;
   - в) максимальная задержка не превышает 180 мс;
   - г) 95 % запросов завершились с ошибкой за 180 мс.

   **Верный ответ: б.** Перцентиль p95 отсекает «хвост» распределения; SLO задают по перцентилям именно потому, что среднее скрывает медленные запросы.

2. Какой вид теста проверяет стабильность сервиса на длительном интервале и выявляет утечки памяти и постепенную деградацию?
   - а) spike; б) stress; в) soak; г) smoke.

   **Верный ответ: в.** Soak-тест держит умеренную нагрузку часами; spike — короткий всплеск, stress — нагрузка выше проектной, smoke — проверка работоспособности.

3. Какой инструмент нагрузочного тестирования использует сценарии на JavaScript с встроенными thresholds и checks?
   - а) Apache Bench (ab); б) k6; в) Locust; г) Prometheus.

   **Верный ответ: б.** У k6 сценарий и пороги описываются на JS; Locust использует Python, `ab` — простой CLI без скриптования, Prometheus — система сбора метрик, а не генератор нагрузки.

4. Что именно проверяет chaos-эксперимент «kill pod» для inference-сервиса в Kubernetes?
   - а) корректность ответов модели;
   - б) способность кластера восстановить сервис (перезапуск пода, перераспределение трафика) без нарушения availability;
   - в) пропускную способность сети;
   - г) размер контейнерного образа.

   **Верный ответ: б.** Удаление пода моделирует внезапный отказ реплики; наблюдаем, как probes, ReplicaSet и балансировка возвращают сервис в строй и как это влияет на SLA по доступности.

5. Зачем при сравнении CPU- и GPU-конфигураций (A/B) считать cost per 1K requests и W/req, а не только p95?
   - а) чтобы уменьшить размер образа;
   - б) чтобы принять решение об оптимизации с учётом trade-off «латентность — стоимость — энергопотребление», а не только скорости;
   - в) это требование Docker;
   - г) чтобы отключить мониторинг.

   **Верный ответ: б.** GPU может дать меньший p95, но оказаться дороже или прожорливее на запрос; обоснованный выбор конфигурации опирается на совокупность метрик.
