# Модуль 1. Архитектура и эксплуатационные требования к промышленным ИИ-системам

> Имя каталога (`M1-Hardware architecture of AI`) историческое и не отражает содержание: модуль посвящён не аппаратной архитектуре, а архитектуре и эксплуатационным требованиям промышленных ИИ-сервисов.

## Место в дисциплине и связь со сквозным проектом

Модуль соответствует разделу 1 РПД и лекции 1. Это стартовая точка дисциплины и первого этапа сквозного проекта («Сервис-контракт и контейнер»). Здесь обучающийся переходит от установки «запустил локально — работает» к постановке «сервис работает в проде и выдерживает нагрузку группы из 30 человек»: разбирается жизненный цикл ИИ-сервиса, границы ответственности эксплуатации и эксплуатационные метрики.

Практический результат ЛР1 — зафиксированный **SLA-контракт** shared ИИ-сервиса на 30 пользователей (целевые SLO по p95, TTFT, throughput, availability). Этот артефакт лежит в репозитории проекта и является отправной точкой для всех последующих модулей: сервис из ЛР2–ЛР3 разворачивается под эти цели, метрики в ЛР6 собираются именно по этим показателям, а в **ЛР8 (Модуль 8)** контракт проверяется данными нагрузочного тестирования — доказывается или опровергается.

Проверяемые индикаторы КРМ (версия 3.0): **LC-5.1** (выбор инструментов и практик промышленной разработки и эксплуатации ИИ-систем) и **BD-5.3** (формулирование эксплуатационных требований и метрик качества). Целевой уровень модуля — **базовый (Б)**.

## Содержание лекции

- **Локальный запуск против production.** Почему «работает на ноутбуке» не равно «работает в проде»: одиночный запрос против конкурентной нагрузки, отсутствие изоляции и лимитов, единая точка отказа, невоспроизводимое окружение, отсутствие наблюдаемости. Ключевой сдвиг дисциплины — от одного пользователя к shared-сервису на 30 человек.
- **Жизненный цикл ИИ-сервиса.** Этапы: упаковка модели → развёртывание → эксплуатация под нагрузкой → мониторинг и алертинг → обновление (rolling update) → вывод из эксплуатации. Инженерная роль эксплуатации и границы ответственности между ML-командой (качество модели) и платформенной командой (доступность, latency, стоимость).
- **SLA / SLO / SLI.** SLI — измеряемый показатель (например, p95 latency); SLO — внутренняя цель по SLI (например, `p95 < 2 c`); SLA — внешнее обязательство перед пользователями с последствиями нарушения. Error budget как связка между целевой availability и допустимым числом сбоев.
- **Метрики эксплуатации ИИ-сервиса.** Латентность p50/p95/p99 (почему цели задают по перцентилям, а не по среднему); для LLM — TTFT (time to first token) и inter-token latency; пропускная способность — TPS (tokens per second) и throughput (RPS); availability (%); экономические и физические метрики — cost per 1K requests и GPU utilization %.
- **Состав production-ИИ-сервиса.** Inference-endpoint (сервер инференса), балансировщик/шлюз доступа с квотами на 30 пользователей, хранилище моделей и артефактов, кэш, реляционная БД метаданных, стек наблюдаемости (метрики, логи, трейсы), контур поставки CI/CD. Как из этих блоков собирается shared-сервис курса.

## Экспресс-тест текущего контроля

**Вопрос 1.** Чем production-эксплуатация ИИ-сервиса принципиально отличается от локального запуска модели?
- A) production всегда быстрее локального запуска
- B) в проде сервис обслуживает конкурентную нагрузку многих пользователей и требует изоляции, лимитов, отказоустойчивости и наблюдаемости
- C) в проде модель обязательно меньше по размеру
- D) локальный запуск не использует Python

*Верный ответ: B.*

**Вопрос 2.** Что означает метрика p95 latency = 2 c?
- A) средняя задержка запроса равна 2 c
- B) максимальная задержка не превышает 2 c
- C) 95 % запросов обслужены быстрее 2 c, а 5 % — медленнее
- D) 95 % запросов завершились с ошибкой

*Верный ответ: C.*

**Вопрос 3.** Как соотносятся SLI, SLO и SLA?
- A) SLI — измеряемый показатель, SLO — внутренняя цель по нему, SLA — внешнее обязательство с последствиями за нарушение
- B) это три названия одного и того же
- C) SLA измеряется, SLO — обязательство, SLI — реклама
- D) SLO задаётся только для CPU, SLA — только для GPU

*Верный ответ: A.*

**Вопрос 4.** Какая метрика LLM-сервиса отражает задержку до появления первого токена ответа?
- A) GPU utilization
- B) TTFT (time to first token)
- C) cost per 1K requests
- D) размер контейнерного образа

*Верный ответ: B.*

**Вопрос 5.** Почему целевые SLO по латентности задают по перцентилям (p95/p99), а не по среднему?
- A) среднее нельзя посчитать
- B) перцентили всегда меньше среднего по определению
- C) среднее скрывает «хвост» медленных запросов, а именно он определяет пользовательский опыт под нагрузкой
- D) так требует Docker

*Верный ответ: C.*
